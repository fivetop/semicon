ORCA 프로젝트 스펙

1. 목적
- ORCA 설계를 Synopsys Design Compiler 기반으로 합성(synthesis)하여 게이트 레벨 netlist 및 제약 파일 생성.

2. 폴더 구조
- 0_Script: 합성 단계별 TCL 스크립트
- 1_Input: RTL, 제약 및 기타 입력 자료
- 2_Output: 합성 결과(언매핑/매핑 DDC, Verilog, SDC 등)
- 3_Log: 실행 로그
- 4_Report: 결과 리포트

3. 작업 흐름
1) 환경 설정
   - `1_run_syn_setup`: `.synopsys_dc.setup` 로드로 DC 환경 구성.
2) RTL 불러오기
   - `2_run_syn_read_verilog`/`2b_run_syn_read_verilog`: 설계 관련 Verilog 파일 분석 및 디자인 세팅.
3) 링크 및 검증
   - `3_run_syn_link_check`: 구조 연결(link) 수행, 체크 리포트 생성, 언매핑 DDC/Netlist 작성.
4) 제약 적용
   - `4_run_syn_constraints`: 타이밍/면적 제약 로드, 불용 셀 정의.
5) 합성 실행
   - `5_run_syn_compile`: 제약에 따라 컴파일 옵션 설정, reset/clock 처리, 최적화 및 scan FF 포함 합성.
6) 리포트 생성
   - `6_run_syn_report`: 클럭, 타이밍, 면적, 제약 위반 등 다양한 리포트 출력.
7) 출력 생성
   - `7_run_syn_write_output`: 매핑 결과 DDC, 게이트 netlist, SDC 파일 작성 및 이름 변경 규칙 적용.

4. 주요 파일
- `:run_syn`: csh 실행 스크립트로 전체 파이프라인을 순차 실행.
- `setup.sh`: 환경 변수 및 모듈을 로드하는 셸 스크립트.
- `main_syn.tcl`: 0_Script 내 스크립트들을 연결하는 메인 TCL.

5. RTL 설계 상세 분석
위치: `2_Synthesis/1_Input/RTL/` (총 31개 Verilog 모듈)

5.1 최상위 설계
- ORCA.v: 최상위 모듈 (파라미터: sd_a_width=10, sd_dq_width=16, pci_data_width=16 등)
- ORCA_TOP.v: Top-level 통합 모듈 (시스템 클록, PCI, SDRAM 인터페이스 포함)

5.2 RISC 프로세서 코어 (16비트)
- RISC_CORE.v: 프로세서 코어 실제 구현
- CONTROL.v: 제어 유닛 (명령어 해석, 제어 신호 생성, 플래그 기반 조건 제어)
- DATA_PATH.v: 데이터 경로 (레지스터-ALU 데이터 흐름 제어)
- ALU.v: 16비트 산술논리 유닛 (add, sub, and, or, xor, shift 등)
  * 입력: Oprnd_A/B(16비트), ALU_OP(6비트)
  * 출력: Lachd_Result(16비트), 플래그(Zero, Negative, Carry)
- REG_FILE.v: 레지스터 파일 (3포트: Read A/B, Write C, 16비트 × 64 레지스터)
- INSTRN_LAT.v: 32비트 명령어 래치

5.3 명령어 처리 (5-단계 파이프라인)
- PRGRM_FSM.v: FSM 상태 관리 (RESET → FETCH → READ_OPS → EXECUTE → WRITEBACK)
- PRGRM_DECODE.v: 분기 명령어 디코더 (조건부 분기 지원)
- PRGRM_CNT_TOP.v: 프로그램 카운터 Top-level
- PRGRM_CNT.v: 8비트 PC 레지스터 (증가, 분기 주소 로드, 리턴 주소 로드)

5.4 스택 메모리 서브시스템
- STACK_TOP.v: Stack 최상위 모듈
- STACK_FSM.v: Stack 상태 관리
- STACK_MEM.v: 스택 메모리 구현 (Push/Pop 동작)

5.5 PCI 인터페이스 (66MHz, 16비트)
- PCI_CORE.v: PCI 핵심 로직 (pidsel, pgnt_n, pframe_n, ptrdy_n, pdevsel_n 신호 처리)
- PCI_RFIFO.v: PCI 읽기 FIFO (32비트, 깊이 32, 비동기 Push/Pop)
  * 상태 플래그: empty, almost_empty, half_full, almost_full, full, error
- PCI_WFIFO.v: PCI 쓰기 FIFO (32비트, 깊이 32)
- PCI_W_MUX.v: PCI 쓰기 데이터 멀티플렉서 (Blender/SDRAM/RISC 데이터 선택)

5.6 SDRAM 인터페이스
- SDRAM_IF.v: SDRAM 인터페이스 모듈 (주소 10비트, 데이터 16비트)
- SDRAM_RFIFO.v: SDRAM 읽기 FIFO (32비트, 깊이 64)
- SDRAM_WFIFO.v: SDRAM 쓰기 FIFO (32비트, 깊이 64)
- SD_W_MUX.v: SDRAM 쓰기 데이터 멀티플렉서

5.7 멀티미디어 가속기
- BLENDER.v: 색상 처리/블렌딩 모듈
  * 입력: 32비트 op1/op2, 4비트 operation
  * 출력: 32비트 result
  * 구현: 3-단계 파이프라인 (s1, s2, s3)
  * 지원: 색상 성분(trans, red, green, blue) 처리
- PARSER.v: 명령어 파서 (4비트 명령어 해석, Blender/Context/FIFO 제어)
- CONTEXT_MEM.v: Context 메모리 (32비트 × 4 레지스터)

5.8 클록 및 전원 관리
- CLOCK_GEN.v: 클록 생성 및 관리 (PLL 기반, 전력 절감 모드 지원)
  * 입력: pclk(66MHz), sdram_clk, sys_clk, test_mode, powersave
  * 출력: o_pclk, o_sdram_clk, o_sys_clk, o_sys_2x_clk
- CLKMUL.v: 클록 멀티플라이어 (NBUFFX4_RVT 버퍼, CLK_1X/2X 출력)
- RESET_BLOCK.v: 리셋 신호 처리 및 동기화
- IOPOWER.v: I/O 전원 관리 (IOVDD, IOVSS, VDD, VSS 분산)

5.9 설계 특성 및 사양
- 아키텍처: 5-단계 파이프라인 RISC 프로세서 + 멀티미디어 가속기(Blender)
- 클록 도메인: sys_clk(125MHz), pclk(66MHz), sdram_clk
- RISC 코어: 16비트 데이터경로, 8비트 명령어
- 외부 인터페이스: PCI(16비트), SDRAM(16비트 × 1K)
- 테스트 지원: Scan, IDDQ, BIST 인터페이스
- 전력 관리: 클록 게이팅, 전력 절감 모드

5.10 모듈 계층 구조
ORCA_TOP (최상위)
├─ RISC_CORE
├─ PRGRM_CNT_TOP → PRGRM_FSM, PRGRM_DECODE, PRGRM_CNT
├─ PCI 인터페이스 → PCI_CORE, PCI_RFIFO, PCI_WFIFO, PCI_W_MUX
├─ SDRAM 인터페이스 → SDRAM_IF, SDRAM_RFIFO, SDRAM_WFIFO, SD_W_MUX
├─ PARSER
├─ BLENDER
├─ CONTEXT_MEM
├─ STACK_TOP → STACK_FSM, STACK_MEM
├─ CLOCK_GEN → PLL, CLKMUL
├─ RESET_BLOCK
└─ IOPOWER


6. 기타 고려사항
- 문서: `DOC/` 폴더에 PROJECT_SPEC.txt(전체 스펙), SCRIPTS_SUMMARY.txt(스크립트 요약), RTL_ANALYSIS.txt(RTL 상세 분석) 보관
- 작업은 원본 `DC_LAB/2_Synthesis` 구조를 그대로 ORCA 하위로 복사해 수행.
- 스크립트 및 설정은 버전 2021.06-SP5-5를 기준으로 함.
- DC_LAB의 전체 폴더(0_Language, 1_Verilog_VCS_Verdi, 3_DFT, 4_ATPG, 5_FM, 6_STA, DC_2016.03, DK, FE, SAED32_EDK, SAED32nm_PDK_04152022)도 ORCA로 복사되어 있음.
- 합성 시 스캔 플립플롭(Scan FF) 포함, 클록 게이팅 적용, 특정 셀(LAS, LAR) 제외.
- 리셋 신호 4개(pci_rst_n, sdram_rst_n, sys_rst_n, sys_2x_rst_n)는 Ideal Network로 처리.

권장: 아님

7.2 CDC 중심 계층화 (권장 1순위) ⭐⭐⭐ - 점수: 70/100
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ✅ Metastability 자동 해결 (Gray 코드 동기화)
  ✅ CDC 검증 자동화 (Synopsys CDC-IFV)
  ✅ 신호 6개로 단순화 (매우 깔끔)
  ✅ 각 도메인 독립 최적화 가능
  ✅ 검증 병렬화 (시간 50% 단축)
  ✅ 합성 효율 높음 (~10분, 병렬)
장점: 안전성 + 현실적 구현 + 검증 자동화
단점: 약간의 추가 학습 필요
개발 시간: 2-3주
권장: ✅ 현재 ORCA에 최적

7.4 방식별 선택 기준
━━━━━━━━━━━━━━━━━━━

현재 상황:
- 설계 완료 (31개 모듈)
- 멀티클록 도메인 (3개)
- Metastability 위험 있음

선택:
1️⃣ 즉시: CDC 중심 계층화 (2-3주 내 시작)
   → 안전성 + 신뢰성 + 검증 자동화
   
2️⃣ 단기: CDC 완성 + 통합 검증
   → 제품 출시 전 Metastability 버그 제거
   
3️⃣ 장기: AXI 마이그레이션 (선택사항)
   → DMA 등 새 Master 추가 시 고려

7.5 비용-편익 분석
━━━━━━━━━━━━━━━━━━

ROI: 높음 ✅

8. 권장 로드맵
==============================================================================

Phase 1: CDC 계층화 (2-3주)

Phase 3: 합성 (1-2주)
- 계층적 합성
- Timing Closure
- 최적화

Phase 4: AXI 준비 (선택, 1주)
- AXI Adapter 아키텍처 설계
- 향후 마이그레이션 로드맵 수립
